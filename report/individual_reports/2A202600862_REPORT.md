### Code Highlights

#### 1. ReAct Loop (`src/agent/agent.py` — hàm `run()`)

Đây là trái tim của hệ thống. Vòng lặp có **circuit-breaker cứng** `max_steps` để ngăn Agent chạy vô hạn:

```python
for step in range(self.max_steps):          # ← hard loop limit (default 6)
    result = self.llm.generate(
        prompt=conversation,
        system_prompt=self.get_system_prompt(),
    )
    llm_output = result.get("content", "")
    conversation += llm_output + "\n"

    # Check Final Answer → break cleanly
    final_answer = _parse_final_answer(llm_output)
    if final_answer:
        break

    # Parse Action → execute tool → append Observation
    action = _parse_action(llm_output)
    if action:
        tool_name, raw_args = action
        observation = self._execute_tool(tool_name, raw_args)
        conversation += f"Observation: {observation}\n\n"
else:
    # Timeout — max_steps exhausted
    final_answer = "I could not complete the task within max steps."
```

**Tại sao dùng `for...else`?** Trong Python, khối `else` của vòng `for` chỉ chạy khi vòng lặp kết thúc **tự nhiên** (không bị `break`). Đây là pattern lý tưởng để phát hiện timeout — nếu Agent tìm được `Final Answer` trước `max_steps` thì `break` và `else` bị bỏ qua.

#### 2. Argument Parser linh hoạt (`_parse_kwargs`)

LLM đôi khi trả về `tool(arg)` (positional) hoặc `tool(key=val, ...)` (keyword). Parser xử lý cả hai:

```python
def _parse_kwargs(raw_args: str) -> Dict[str, str]:
    # Try key=value pairs first
    kv_pairs = re.findall(r"(\w+)\s*=\s*([^,]+)", raw_args)
    if kv_pairs:
        return {k.strip(): v.strip().strip("'\"") for k, v in kv_pairs}
    # Fallback: single positional argument
    return {"_arg0": raw_args.strip().strip("'\"")}
```

#### 4. System Prompt với Format Enforcement

```

  Thought: <reasoning>
  Action: tool_name(arg1_name=arg1_value)
  Observation: <do NOT write this yourself>
  Final Answer: <answer when done>

```

Việc ghi rõ **"Do NOT write Observation yourself"** giải quyết lỗi phổ biến nhất: LLM tự bịa ra Observation thay vì chờ kết quả tool thực.

```

    │
```

## II. Debugging Case Study (10 Points)

### Problem: LLM sử dụng single-quote trong `item_name` khiến parser nhận sai value

**Bối cảnh**: Khi chạy TC-03 (mua 2 iPhone 15), Agent gọi:
```
Action: get_product_price(item_name='iPhone 15')
```

**Log thực tế** (từ `logs/2026-06-01.log`, event `AGENT_TOOL_CALL`):
```json
{
  "step": 1,
  "tool": "get_product_price",
  "raw_args": "item_name='iPhone 15'"
}
```

**Vấn đề**: Nếu không xử lý, `_parse_kwargs` sẽ trả về `{'item_name': "'iPhone 15'"}` (cả dấu nháy đơn bao gồm trong value), khiến lookup trong `PRICE_DB` thất bại vì không có key `"'iphone 15'"`.

**Chẩn đoán**: Regex `([^,]+)` bắt được cả dấu `'` bao quanh giá trị. LLM không tuân theo hoàn toàn format "no quotes" mà tự thêm vào.

**Giải pháp**: Thêm `.strip("'\"")` sau khi extract value:

```python
# Trước (bị lỗi):
result[k.strip()] = v.strip()

# Sau (đã fix):
result[k.strip()] = v.strip().strip("'\"")   # ← strip cả single và double quotes
```

**Kết quả**: Agent chạy TC-03 thành công với `_parse_kwargs` nhận `{'item_name': 'iPhone 15'}` (không có quotes). Tool lookup tìm đúng key trong PRICE_DB.

**Bài học**: LLM không bao giờ hoàn toàn tuân theo format. Parser phải **defensive** — normalize input trước khi xử lý.

## III. Personal Insights: Chatbot vs ReAct Agent (10 Points)

### 1. Reasoning — `Thought` block giúp Agent như thế nào?

Chatbot trả lời **ngay lập tức** từ knowledge được train, nên:
- Không biết giá hiện tại → đoán mò ($799 USD thay vì 25,000,000 VND)
- Không biết tồn kho → nói chung chung "check retailer"
- Không thể tính toán đa bước có dữ liệu thực

Agent dùng `Thought` để **lập kế hoạch** trước khi hành động:
```
Thought: I need to find the price first, then check stock,
         then get the discount, then calculate shipping.
Action: get_product_price(item_name=iPhone 15)
```
`Thought` buộc LLM phân rã vấn đề phức tạp thành các bước nhỏ có thể thực thi. Đây là điểm mấu chốt — không phải LLM thông minh hơn, mà là **kiến trúc cho phép LLM tiếp cận thông tin thực tế**.

### 2. Reliability — Khi nào Agent tệ hơn Chatbot?

| Tình huống | Chatbot | Agent |
| :--- | :--- | :--- |
| Simple Q&A không cần data | ✅ Nhanh (1,227ms) | ⚠️ Chậm (2,568ms) — gọi thêm LLM turn |
| Câu hỏi chung, không cần tool | ✅ Trả lời ngay | ❌ Có thể gọi tool không cần thiết |
| Multi-step với data thực | ❌ Hallucinate | ✅ Chính xác 100% |

**Kết luận**: Agent vượt trội ở multi-step reasoning với real data. Chatbot vẫn thắng ở câu hỏi đơn giản về **tốc độ** và **chi phí token**. Trong production, nên dùng router để quyết định khi nào cần Agent.

### 3. Observation — Feedback vòng lặp ảnh hưởng thế nào?

TC-04 (coupon FAKE99) là ví dụ điển hình. Sau khi nhận Observation:
```
Observation: Coupon 'FAKE99' is invalid or expired.
```

Thay vì dừng lại, Agent tiếp tục:
```
Thought: The coupon is invalid, but the user still wants to buy.
         I should check stock and price, then calculate without discount.
Action: check_stock(item_name=AirPods Pro)
```

Đây là **adaptive behavior** — Agent điều chỉnh kế hoạch dựa trên kết quả thực tế, không chỉ đơn thuần thông báo lỗi. Chatbot trong cùng trường hợp chỉ nói "thử apply và xem sao" — không có khả năng recover.

## IV. Future Improvements (5 Points)

### Scalability — Xử lý song song và async

Hiện tại mỗi tool call là synchronous. Trong production với nhiều tool cùng có thể chạy song song:
```python
# Thay vì gọi tuần tự:
price = get_product_price("iPhone 15")
stock = check_stock("iPhone 15")

# Có thể dùng asyncio để chạy song song:
price, stock = await asyncio.gather(
    async_get_product_price("iPhone 15"),
    async_check_stock("iPhone 15"),
)
```
Điều này giảm latency đáng kể cho các task có thể parallelize.

### Safety — Supervisor LLM + Guardrails

Thêm một LLM thứ hai đóng vai "Supervisor" kiểm tra output của Agent trước khi trả cho người dùng:
- **Input sanitization**: Kiểm tra tool arguments không có injection (SQL injection, prompt injection).
- **Output validation**: Xác nhận số tiền tính toán hợp lý (không âm, không quá lớn bất thường).
- **Max cost guard**: Nếu tổng `cost_estimate` trong session vượt ngưỡng → reject query.

### Performance — Vector DB cho Tool Retrieval

Khi hệ thống mở rộng lên 50+ tools, đưa toàn bộ tool descriptions vào system prompt không hiệu quả (tốn token). Giải pháp:
1. Embed tất cả tool descriptions vào **vector database** (FAISS / Pinecone).
2. Với mỗi user query, **semantic search** để chỉ lấy top-K tools liên quan nhất.
3. Chỉ đưa top-K tools vào prompt → giảm 60-80% prompt tokens.

### Production-grade: Chuyển sang LangGraph

Kiến trúc hiện tại là linear loop. Để hỗ trợ các workflow phức tạp hơn:
- **Conditional branching**: Tool A thành công → Tool B, thất bại → Tool C.
- **Multi-agent**: Một Orchestrator Agent điều phối các Sub-agents chuyên biệt (SearchAgent, CalculatorAgent).
- **Persistence**: Lưu state của Agent vào DB để resume task bị gián đoạn.
